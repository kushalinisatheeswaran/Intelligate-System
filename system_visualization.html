🔑 JWT Authentication Loop

All client routes utilize stateless JSON Web Tokens. The credentials flow initiates on user login, signing the token with role metadata (Student or Gate Officer), which is verified by API gateway middleware for subsequent verification or manual gate overrides.

📡 WebSockets & FCM

Flask-SocketIO delivers real-time notifications to the Officer's dashboard when vehicles trigger verification, while Firebase Cloud Messaging (FCM) issues high-priority push notifications to standby mobile clients when the app is backgrounded.

⚙️ Architectural Adaptation: Proposed vs. Implemented Stack

To support the intensive computational loads of OpenCV and OCR models, the development team adapted the initial web stack. Python was chosen as the core language for backend processes to facilitate native access to computer vision libraries, and the database was structured around a high-performance relational framework.

📋 Theoretical / Initial Goals
  • Frontend UI: React / React Native client structure
  • Backend Server: Node.js + Express API setup
  • Authentication: Client-side Firebase Auth integration
  • Database Layer: Firestore (NoSQL Document Collections)
🚀 Codebase Implementation
  • Frontend UI: Expo Mobile App (React Native + SecureStore)
  • Backend Server: Python Flask + Flask-SocketIO (eventlet mode)
  • Authentication: API-level JWT (Bearer Tokens) with claims
  • Database Layer: SQLAlchemy relational models (Postgres / SQLite)
  • Edge Computing: OpenCV license detection + easyOCR pipeline

📅 6 Development Phases Timeline

Systematic workflow mapping milestones from foundational APIs to final validation testing.

Phase 1: Backend Setup & Dummy APIs Completed

Initial creation of backend structures, Flask application skeleton, CORS policies, environment variables configuration, and health check routes.

  • Setup of Flask application factory structure inside `app/__init__.py`.
  • Configuration of dummy responses in `/api/verify` for testing mock hardware signals.

Phase 2: Auth Integration Completed

Development of token-based authentication mechanism. Creating security policies, encrypting credentials, and protecting endpoints.

  • Integration of `flask-jwt-extended` for secure verification signatures.
  • Protected API endpoints requiring Bearer tokens in headers.
  • Client-side SecureStore setup to persist sessions.

Phase 3: Database Design Completed

Definition of relational SQL tables. Writing DB schemas representing access history logs, registered vehicles, student accounts, and reviewer queues.

  • Design of User, Vehicle, StudentID, AccessLog, and Pending tables.
  • Configured Flask-SQLAlchemy migrations and database seed files.

Phase 4: Gate Access Logic Completed

Core logic verification rules. Creating pathways to identify scanned plates or barcodes against database tables, handling grant/denial routing.

  • Check of plate/barcode records via relational entity mappings.
  • Creation of PendingApprovals items for unknown vehicles.
  • Real-time websocket events trigger using Socket.IO server.

Phase 5: Logs & Notifications Completed

Building the notification pipeline. Setting up FCM service integrations for mobile notification triggers and database activity log recording.

  • FCM module built to parse service account JSON keys.
  • Trigger of high-priority push events for unrecognized vehicles.
  • Persistent logging of all entries and exists with direction metadata.

Phase 6: Integration & Hardware Testing Testing

Connecting software components with physical hardware. Hooking ESP32 over serial ports or HTTP requests, and calibrating ANPR cameras.

  • ESP32 communication via serial commands and HTTP webhook routes.
  • Wiring of gate servo control and status indicator lights.
  • Real-world field validation of CV license capture accuracy.
